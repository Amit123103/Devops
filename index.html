Unit I: Basics of DevOps Infrastructure

Explore Linux OS-level virtualization through kernel primitives like Namespaces and Control Groups (cgroups), Docker engine internals, and the Docker CLI.

Linux Namespaces

Enforces process boundary isolation. Limits process visibility across network, mounts, process tree (PID), and user IDs.

Cgroups (Control Groups)

Monitors and restricts host compute resource distribution (CPU cycles, memory allocation, block I/O) to containers.

Process Virtualization

Unlike bare-metal or hypervisor-bound Virtual Machines, containers use the host's Linux kernel natively for speed and density.

Core Commands & Demo

# Pulling basic Nginx web server image
docker pull nginx:alpine

# Run an isolated web server container
docker run -d --name amit_web_server -p 80:80 nginx:alpine

# Checking operational containers
docker ps -a

Unit II: Image Building & Container Management

Advanced Dockerfile optimization strategies. Understand read-only layered caching, storage volumes, bind mounts, and isolated bridge networking.

Read-Only Layers

Dockerfile commands create stacked read-only file system layers. Optimizes storage through Copy-on-Write (CoW).

Docker Volumes

A designated persistent layer stored in host directories, managed exclusively by the Docker daemon.

Bind Mounts

Binds a strict path from the host directly into the container. Excellent for live code reloading in dev stages.

Dockerfile Implementation Example

# Use secure, lightweight base image
FROM node:20-alpine
LABEL maintainer="amit@example.com"
WORKDIR /usr/src/app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

Unit III: Microservices with Docker Compose

Designing decoupled, resilient enterprise architectures. Master multi-container orchestration with Docker Compose YAML configuration.

Decoupled Scaling

Isolating monoliths into small independent services behind an API gateway for microservice fault isolation.

Service Orchestration

Docker Compose allows creating services, networks, volumes, environment definitions, and secrets with a single script.

Dependencies Control

Use depends_on field constraints to control the startup order sequence of interconnected backend servers.

WordPress & MySQL Compose YAML

version: '3.8'
services:
  db:
    image: mysql:8.0
    restart: always
    environment:
      MYSQL_ROOT_PASSWORD: root_secret_pwd
      MYSQL_DATABASE: wordpress
    volumes:
      - db_data:/var/lib/mysql
  wordpress:
    image: wordpress:latest
    ports:
      - "8080:80"
    depends_on:
      - db
    volumes:
      - app_data:/var/www/html
volumes:
  db_data:
  app_data:

Unit IV: Maven Build Automation

Master Project Object Model configurations, strict transitive dependency scoping, Maven plugins, and Dockerized multi-stage deployments.

POM Structure

The pom.xml explicitly controls external artifact fetching, compiler constraints, and output bundling configurations.

Transitive Dependency

Maven builds resolve dependency versions dynamically using local-nearest definitions to mitigate classpath pollution.

Multi-Stage Containers

Builds the heavy application source code within a build image, then injects binaries directly into thin JRE runtimes.

Maven Multi-Stage Dockerfile

# Stage 1: Dependency resolution & package
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn clean package -DskipTests

# Stage 2: Runtime Container Deployment
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Unit V: Continuous Integration with GitHub Actions

Leverage modern workflow files in your codebase to run tests and push production-ready images on commits.

Workflow Automations

Listen directly to repository webhook triggers like push or pull_request to kickstart automation tests.

Parallel Runners

Utilize multi-job pipelines in dedicated cloud runners (Ubuntu, Windows) to run tests parallelly.

Matrix Builds

Test matching variations simultaneously across multiple Node or JDK platforms within a single job execution block.

GitHub Actions Workflow Template

name: Continuous Integration Suite
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-java@v4
      with:
        java-version: '17'
        distribution: 'temurin'
    - name: Execute JUnit Automation Tests
      run: mvn test

Unit VI: CI/CD with Jenkins

Establish robust declarative multi-stage software delivery pipelines utilizing persistent Master and Agent worker topologies.

Master/Agent Architecture

Master orchestrates job scheduling, triggers, and configurations. Ephemeral agents execute steps securely.

Declarative Syntax

Structured, versioned Groovy DSL configuration saved within code as a Jenkinsfile to improve reliability.

Dynamic Docker

Spin up container execution layers natively within stages so each step runs inside its isolated build environment.

Declarative Jenkinsfile Configuration

pipeline {
    agent any
    stages {
        stage('Checkout') {
            steps { git branch: 'main', url: 'https://github.com/Amit/Devops.git' }
        }
        stage('Build & Test') {
            steps { sh 'mvn clean test' }
        }
        stage('Docker Push') {
            steps {
                sh 'docker build -t amit/app:latest .'
                sh 'docker push amit/app:latest'
            }
        }
    }
}
