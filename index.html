Unit I: Basics of DevOps Infrastructure

An analytical approach to container engines. Understand how hardware resources are abstracted natively on the Linux Kernel via Namespaces, cgroups, and storage layer controls.

Origins of Virtualization

Heavy hypervisor abstraction requires duplicate guest kernel installations. Modern container engines bypass guest OS layers entirely, executing processes natively on the host kernel.

Kernel Namespaces

Creates explicit logical boundary boundaries. Implements isolation across Network Interfaces, Mount points, Inter-Process Trees (PID), and User/UID registries.

Cgroups V1 & V2

Acts as process throttlers. Restricts usage of hardware compute resources: hard/soft RAM limitations, block device read-write access limits, and CPU allocation shares.

Deep Dive: Namespace Isolation Specifics

Namespaces alter how processes view the system. For example, PID namespace assigns a child process as PID 1 inside the container, while it retains a high-order PID on the host. NET namespace encapsulates physical interfaces, routing tables, and firewalls into isolated interfaces.

Terminal Playground Session
# Determine running system properties & namespaces
lsns -t pid

# Fetch Nginx container image to local store
docker pull nginx:alpine

# Launch detached web server with host port mapping
docker run -d --name amit_web_server -p 8080:80 nginx:alpine

# Inspect low-level process and kernel metadata
docker inspect amit_web_server

Unit II: Image Building & Container Management

Designing lean, high-fidelity images. Learn the mechanics of Copy-on-Write storage layers, .dockerignore contexts, and explicit networking topology designs.

Storage Layer Mechanics

Docker images are read-only filesystem stacks. Modifications create thin, dynamic read-write layers via Copy-on-Write (CoW).

Persistence Architecture

Decouples storage. Volumes persist outside container lifecycles; Bind Mounts explicitly map host paths to containers.

Context Filtering

Using .dockerignore stops non-essential files from migrating into the Docker Daemon's working context memory.

Understanding Storage Management Options

When running database services like MySQL or Redis, standard ephemeral storage is unviable. Using -v amit_volume:/var/lib/mysql ensures your data persists even if the container is removed.

Production Dockerfile Blueprint
# Use secure, ultra-thin alpine base image
FROM node:20-alpine

# Set application labels
LABEL maintainer="amit@example.com"
LABEL version="1.0.0"

# Setup secure working context
WORKDIR /usr/src/app

# Leverage cached dependency installation layer
COPY package*.json ./
RUN npm ci --only=production

# Copy functional assets
COPY . .

# Expose server ingress port
EXPOSE 3000

# Set entry point execution command
CMD ["node", "server.js"]

Unit III: Microservices with Docker Compose

Managing multi-tier microservices stacks using Docker Compose. Scale and interconnect backend services using declarative YAML topologies.

Application Decomposition

Converts single application monoliths into independent microservices to improve horizontal scalability and fault isolation.

Declarative Topology

Defines complex services, private networks, volumes, and environment variables within a unified YAML file configuration.

Dependency Ordering

Uses the depends_on directive to control startup sequences for interconnected servers (e.g., ensuring a database starts before the web server).

Architecture Specifics: API Gateway Integration

An API Gateway acts as the entry point for clients, routing requests to appropriate backend services. This architecture decouples clients from service changes and provides unified authentication and logging.

Multi-Service Stack Composition YAML
version: '3.8'

services:
  db:
    image: mysql:8.0
    container_name: amit_mysql_service
    restart: always
    environment:
      MYSQL_ROOT_PASSWORD: root_secret_pwd
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wordpress_usr
      MYSQL_PASSWORD: wordpress_pwd
    volumes:
      - wordpress_db_data:/var/lib/mysql
    networks:
      - internal_network

  wordpress:
    image: wordpress:latest
    container_name: amit_wordpress_service
    restart: always
    ports:
      - "8080:80"
    environment:
      WORDPRESS_DB_HOST: db:3306
      WORDPRESS_DB_USER: wordpress_usr
      WORDPRESS_DB_PASSWORD: wordpress_pwd
      WORDPRESS_DB_NAME: wordpress
    depends_on:
      - db
    volumes:
      - wordpress_app_data:/var/www/html
    networks:
      - internal_network

volumes:
  wordpress_db_data:
    driver: local
  wordpress_app_data:
    driver: local

networks:
  internal_network:
    driver: bridge

Unit IV: Maven Build Automation

Optimize software packaging using the Maven lifecycle. Learn about transitive dependency resolution and multi-stage container optimization.

POM Architecture

The pom.xml file configures external library fetching, compiler targets, and build goals.

Dependency Resolution

Maven builds resolve dependency trees automatically. Uses nearest-definition rules to avoid classpath version conflicts.

Multi-Stage Containers

Separates build and runtime environments. Builds binaries in a Maven image and copies them to a thin runtime JRE image.

Understanding the Maven Lifecycle

Maven divides the build process into distinct phases. Common phases include validate, compile, test, package, verify, install, and deploy. Executing a phase triggers all preceding phases automatically.

Multi-Stage Java Dockerfile
# Stage 1: Build the Maven application
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn clean package -DskipTests

# Stage 2: Runtime Container
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

Unit V: Continuous Integration with GitHub Actions

Implement native workflow automations inside GitHub. Learn how to trigger test phases and push production-ready images on code commits.

Native Workflow Scripts

Listens for direct source events like push or pull_request to trigger automated test suites.

Infrastructure Runners

Uses multi-job workflows in dedicated cloud runners (Ubuntu, macOS, Windows) to run tests in parallel.

Matrix Builds Strategy

Tests matching variations across multiple platforms or SDK versions within a single job execution block.

Understanding Runners and Marketplace Integrations

Workflows run on isolated virtual machines. Using pre-packaged marketplace actions like actions/checkout and actions/setup-java simplifies writing workflows.

Production Workflow Script (.github/workflows/ci.yml)
name: CI/CD Multi-Job Integration

on:
  push:
    branches: [ "main" ]
  pull_request:
    branches: [ "main" ]

jobs:
  build_and_test:
    name: Build & Run Tests
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: Unit_04_Maven_Automation

    steps:
    - name: Checkout Code from Repo
      uses: actions/checkout@v4

    - name: Set up JDK 17
      uses: actions/setup-java@v4
      with:
        java-version: '17'
        distribution: 'temurin'
        cache: maven
        cache-dependency-path: '**/pom.xml'

    - name: Run Maven Tests
      run: mvn test

    - name: Package Maven Artifact
      run: mvn package -DskipTests

Unit VI: CI/CD with Jenkins

Learn declarative multi-stage software delivery pipelines. Understand persistent Master and Agent worker topologies and dynamic container execution.

Master/Agent Clustering

The Master orchestrates job scheduling and triggers. Agents execute pipeline steps inside isolated compute layers.

Declarative Pipeline DSL

A structured, versioned Groovy DSL saved inside the repository as a Jenkinsfile to improve reliability.

Dynamic Agent Layers

Run stages natively inside temporary container execution environments to isolate build dependencies.

Understanding Webhooks & Automation Triggers

Webhooks trigger automated build jobs in Jenkins on every code commit. Alternatively, the pollSCM directive checks the repository periodically for changes.

Declarative Jenkinsfile Configuration Blueprint
pipeline {
    agent any

    environment {
        IMAGE_NAME = "amit/my-app"
        IMAGE_TAG = "latest"
    }

    stages {
        stage('Checkout Code') {
            steps {
                git branch: 'main', url: 'https://github.com/Amit123103/Devops.git'
            }
        }

        stage('Compile & Test') {
            steps {
                sh 'mvn clean test'
            }
        }

        stage('Build & Package') {
            steps {
                sh 'mvn package -DskipTests'
            }
        }
    }
}
