Unit I: Basics of DevOps Infrastructure

Understand hypervisor abstraction, namespaces, cgroups, and isolated CLI controls completely.

Simulation Console

Core Concepts & Theory:

Before containers, applications were deployed directly on physical hardware or Virtual Machines (VMs). Virtual Machines sit above hardware via hypervisor to run guest instances, incurring large CPU overhead. Containers provide operating-system-level virtualization, sharing a single host kernel while keeping individual files, environment variables, and processes completely isolated.

Process Isolation Mechanisms:

Linux utilizes Namespaces to handle complete logical isolation of processes, networking, and inter-process variables. Control groups (cgroups) manage complete physical hardware allocation and constraint boundaries across processing components.

📋 Essential Command Pipeline

# Verify docker status
docker version

# Pull an image completely from Registry
docker pull nginx:latest

# Spin up detached web application
docker run -d -p 8080:80 --name amit_web_server nginx:latest

# List all active containers
docker ps

Image Layering & Copy-on-Write:

Images are immutable read-only sequence layers where each represents a specific Dockerfile step instruction. Copy-On-Write (CoW) copies required base assets up into the writeable top layer before making any modifications, optimizing file sizes.

Docker Storage Models:

Volume directories are completely managed and lifecycle-controlled by the container engine. Bind mounts bind external host folders directly into the processing environment.

📝 Recommended Dockerfile & Workspace Layout

FROM node:20-alpine
WORKDIR /usr/src/app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

Monolithic vs Microservices:

Microservices divide architectures into small standalone units that communicate over APIs, enhancing scalability and fault tolerance. Docker Compose orchestrates these units in isolated network environments via single commands.

📄 Multi-Container Deployment Blueprint (`docker-compose.yml`)

version: '3.8'
services:
  db:
    image: mysql:8.0
    restart: always
    environment:
      MYSQL_ROOT_PASSWORD: root_secret_pwd
      MYSQL_DATABASE: wordpress
    volumes:
      - wordpress_db_data:/var/lib/mysql
    networks:
      - internal_network
  wordpress:
    image: wordpress:latest
    ports:
      - "8080:80"
    depends_on:
      - db
    networks:
      - internal_network

volumes:
  wordpress_db_data:
networks:
  internal_network:

Continuous Integration Lifecycle:

Continuous integration automatically aggregates codebase updates via workflows, triggers isolated runners, packages target binaries, and enforces software delivery standards completely.

📦 Standard Workflow YAML Definition (`.github/workflows/ci.yml`)

name: Complete CI Pipeline
on:
  push:
    branches: [ main ]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with:
          java-version: '17'
          distribution: 'temurin'
      - run: mvn test

Establish distributed Worker-based pipelines completely using Groovy steps.

Simulation Console

Master/Worker Architecture:

Jenkins establishes reliable pipelines via Master and Worker Agent connections. This divides orchestrations across nodes to balance server workloads and speed up testing cycles.

🤖 Declarative Pipeline Template (`Jenkinsfile`)

pipeline {
    agent any
    stages {
        stage('Checkout Code') {
            steps {
                git branch: 'main', url: 'https://github.com/Amit123103/githubdevops.git'
            }
        }
        stage('Compile & Test') {
            steps {
                sh 'mvn clean test'
            }
        }
    }
}
